# 能力边界与运维注意

本文档说明当前 Kubo + IPFS Cluster 私有化方案**做了什么、不做什么**，以及由此带来的数据模型、冗余/备份与安全注意事项。用于评估方案是否满足业务诉求。

- 方案：基于 IPFS Cluster（CRDT）的私有化内容托管，可单节点起步、按需扩为多节点多副本。
- 部署形态：[单机集群部署](./SINGLE_HOST_DEPLOYMENT.md)（本机三容器试验）、[多机部署](./MULTI_HOST_DEPLOYMENT.md)（ECS/EC2，含单节点模式）。

---

## 1. 能力边界

本方案本质是一个**内容寻址的对象存储 + HTTP 网关**：Agent 上传内容拿到 CID，浏览器通过网关按 CID 访问并渲染。

### 1.1 已具备

- Agent 通过 cluster 代理 `POST /api/v0/add` 上传 HTML，拿到内容寻址链接（CID）。
- 浏览器访问 `:8080/ipfs/<CID>` 直接渲染 HTML（含目录 + 相对资源）。
- 数据仅存私有网络、不进入公网 IPFS（`swarm.key` 私有网隔离）。
- 重启不丢数据（卷持久化）。
- **多副本与故障容忍**：CID 在多个节点各存一份，挂掉部分节点不影响读取（单节点模式无此项，见 §3）。

### 1.2 暂不具备（需上层补齐或后续演进）

| 能力 | 现状 | 说明 |
|------|------|------|
| 友好短链 / 固定路径 | ❌ | 链接是一长串 CID。需加一个薄映射服务（短码→CID）或反代重写 |
| 访问鉴权 / 谁能看 | ❌ | 网关默认无鉴权，**任何拿到 CID 的人都能访问**，需在反代层加 auth |
| 权限隔离（A 看不到 B 的） | ❌ | CID 即凭证，无账户体系，须在上层自建 |
| 公网 / TLS / 域名 | ❌ | 需反向代理 + 证书 |

**一句话定位**：IPFS 负责"存储 + 寻址 + 渲染 + 多副本"，**不负责"账户 + 权限 + 短链"**。后三者要么自建一层薄服务，要么换方案。

> ⚠️ 若核心诉求是"权限可控的内部分享"，需注意 IPFS 的内容寻址模型（无 ACL、CID 即明文凭证）与该诉求是逆风的，应认真评估是否非它不可。

## 2. 数据模型：pin、临时文件与 GC

IPFS 中所有数据以 block 形式存在本地仓库（容器内 `/data/ipfs`），分两类：

- **被 pin 的内容**：永久保留，垃圾回收（GC）**不会**删除。经 cluster 上传（`pin=true`）的内容由 cluster 在各节点 pin 住。
- **未 pin 的内容（临时）**：例如读取/缓存产生的块，以及**过期被 cluster 自动 unpin 的内容**。本部署的 kubo 以 `--enable-gc` 启动：按 `Datastore.GCPeriod`（默认 1h）周期回收，仓库达 `StorageMax`（默认 10GB）九成水位也会触发；也可手动 `docker exec <kubo容器> ipfs repo gc`。

**结论**：Agent 经 cluster 代理上传，内容被全集群 pin，不会被 GC 误删；到期 unpin 后，块在各节点下一轮 GC 时回收——**"过期"与"链接实际不可读"之间最多相差一个 GC 周期（默认 ≤1h）**。

## 3. 冗余与恢复

关于 IPFS "多副本 / 漂移 / 恢复" 的常见误解，逐点澄清：

- **标准 IPFS 不自动多副本**：一个 CID 只存在于**主动 pin 了它的节点**上。其它节点访问时只是按需缓存（临时，可被 GC）。多副本要靠 **IPFS Cluster** 编排（本方案即用它）。
- **不会漂移**：标准 IPFS 无自动数据迁移/再平衡。CID 是内容哈希，内容变了 CID 就变，不存在"同一 CID 内容漂移"。
- **Cluster 提供的恢复**：cluster 维护全局 pinset 并按副本因子分配；节点故障时可在其它节点补 pin（多机生产形态）。

**单节点模式（只跑 1 个节点）**：

- 数据 **100% 只存在这一个节点的卷里，无任何副本**——退化为单点，无冗余、无容灾。
- 适合开发/小规模；**必须自建备份**：定期备份宿主机 `./runtime/data/ipfs`（或多机部署的 `DATA_DIR`），或对存储卷做快照。

> 注意：多副本防**硬件故障**，但**不防误删/逻辑错误**——cluster 删 pin 会全集群同步删除。无论几节点，备份都不可省。

## 4. 安全现状与生产化清单

### 4.1 当前安全现状：可信局域网/VPC 内裸奔

- 🔴 **控制面端口 = 完全控制权**：kubo API `5001`、cluster REST `9094`、cluster 代理 `9095` 均无鉴权 + CORS 全开（`Allow-Origin: *`）。任何可达者都能上传、删 pin、改配置。**绝不可暴露到公网**。
- 🟠 **8080 网关无鉴权**：任何拿到 CID 的人都能读，无"私有内容"概念。CID 泄露 = 内容泄露。
- 🟠 **同源 XSS 风险**：path 网关下不同 CID 共享同一 origin（`host:8080`），恶意 HTML 可读同 origin 下其它内容的 localStorage/cookie。

扎实的隔离：`swarm.key` 私有网络 + 清空公网 bootstrap → 不接触公网 IPFS；多机时 `4001/9096` 仅在 peer 间开放。

### 4.2 生产化安全清单（按优先级）

1. **控制面端口（5001/9094/9095）永不对用户侧/公网开放**——只让 Agent 网段可达，防火墙/ACL/安全组锁死。
2. **8080 网关前套反向代理 + 鉴权**（SSO / Token），不要让网关裸奔。
3. **切 subdomain 网关**（`<CID>.ipfs.host`）隔离 origin，防 XSS。
   - 注意：本方案为 `localhost` 关闭了子域名重定向（见 `scripts/init-cluster.d/001-config.sh`，便于命令行/本机访问）；生产应为业务域名在 `Gateway.PublicGateways` 下反向开启子域名隔离（不影响 localhost）。
4. **收紧 CORS**：白名单具体来源，不要 `*`。
5. **TLS 终结在反代**。
6. **数据卷定期备份**（呼应 §3）。
7. **`CLUSTER_SECRET` 与 `swarm.key` 是集群信任根**，按密钥严格保管与分发。

## 5. 后续演进方向

- 友好短链：薄映射服务（短码→CID）或反代重写。
- 鉴权与权限：反向代理 + SSO/Token，上层账户体系。
- 公网访问：反向代理 + 域名 + TLS + subdomain 网关。
- K8s 形态：StatefulSet + PVC（留作后续，见 [多机部署](./MULTI_HOST_DEPLOYMENT.md) §8）。
